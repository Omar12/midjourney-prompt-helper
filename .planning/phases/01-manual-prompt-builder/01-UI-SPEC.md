# Phase 1 — UI Design Contract

> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

## Spacing Scale

All values are multiples of 4. Used via Tailwind utility classes (`p-2`, `gap-4`, etc.)
and shadcn/ui component defaults — not via custom CSS variables.

| Token | Value | Tailwind class | Primary usage |
|-------|-------|----------------|---------------|
| xs | 4px | `gap-1`, `p-1` | Icon-to-label gaps, chip internal padding |
| sm | 8px | `gap-2`, `p-2` | Chip-to-chip spacing, button padding |
| md | 16px | `gap-4`, `p-4` | Default element spacing, input padding |
| lg | 24px | `gap-6`, `p-6` | Pane inner padding, section separation |
| xl | 32px | `gap-8`, `p-8` | Layout gaps between major UI regions |
| 2xl | 48px | `gap-12` | Vertical rhythm between controls pane sections |
| 3xl | 64px | `gap-16` | Not used in Phase 1 |

**Exceptions:**

- Chip ✕ remove button: minimum 44×44px touch target (WCAG 2.5.5 AA). Achieved via
  `size="icon"` variant on shadcn/ui `Button` with explicit `min-w-[44px] min-h-[44px]`
  if the default falls short. Do not reduce below 44px.
- Preview pane right-column width: fixed at `w-96` (384px) on wide viewports. This is a
  layout value, not a spacing token.

## Typography

Three roles render in Phase 1. Exactly 2 font weights in use: 400 (regular) and 500 (medium).

| Role | Size | Weight | Line Height | Tailwind class | Usage |
|------|------|--------|-------------|----------------|-------|
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-relaxed` | Intent textarea content, chip labels, preview prompt text |
| Label | 14px | 500 (medium) | 1.4 | `text-sm font-medium leading-snug` | Input labels, section headings ("Preview"), chip ✕ aria-label, button text |
| Caption | 12px | 400 (regular) | 1.4 | `text-xs font-normal leading-snug` | Character count hint, helper text below inputs |

**Future phases (not rendered in Phase 1):** Heading 20px/600/1.2 reserved for later-phase nav; not rendered in Phase 1.

**Constraints:**
- Body at 1.5 line-height ensures the intent textarea and preview text are comfortable
  for multi-line reading — Midjourney prompts are often 50–200 characters.
- Preview text renders in `font-mono` (Body size, regular weight) to distinguish the
  assembled prompt string from prose. This is the single exception to the sans-serif default.

## Color

Dark-mode values. All colors are declared as shadcn/ui CSS variables; the hex values
below are representative for design reference only. The executor uses the CSS variable
names, not hardcoded hex.

| Role | CSS variable | Representative hex | OKLCH approx | Usage |
|------|-------------|-------------------|--------------|-------|
| Dominant (60%) | `--background` | `#09090b` | oklch(9% 0.01 286) | Main page background, both panes |
| Secondary (30%) | `--card` / `--muted` | `#18181b` | oklch(14% 0.01 286) | Controls pane surface, preview pane surface, chip background |
| Accent (10%) | `--primary` | `#7c3aed` | oklch(54% 0.21 291) | Copy button (primary action only), active state indicator |
| Destructive | `--destructive` | `#dc2626` | oklch(53% 0.22 27) | ClearDialog confirm action button only |
| Foreground | `--foreground` | `#fafafa` | oklch(98% 0 0) | Body text, labels |
| Muted foreground | `--muted-foreground` | `#71717a` | oklch(53% 0.01 286) | Placeholder text, caption text, empty state body |
| Border | `--border` | `#27272a` | oklch(19% 0.01 286) | Divider between panes, input borders, chip borders |

**Base palette:** shadcn/ui zinc dark theme. Accent overridden to violet-700 (`#7c3aed`)
— zinc grays alone are insufficient to draw the eye to the primary CTA.

**Accent reserved for:**
1. Copy button (the single primary call-to-action)
2. Nothing else in Phase 1

Phase 2 may extend accent to version-selector active state. Do not apply accent to
chip hover states, intent input focus rings, or decorative elements.

**Focus ring:** Use shadcn/ui default `--ring` variable (a lighter violet tint derived
from accent). Do not suppress focus outlines — keyboard accessibility is required.

## Layout Contract

### Two-Pane (D-09, locked)

```
┌─────────────────────────────────────────────────────────┐
│                    App (h-screen flex)                   │
│                                                          │
│  ┌─────────────────────────┐  ┌──────────────────────┐  │
│  │   Controls pane          │  │  Preview pane         │  │
│  │   flex-1 overflow-y-auto │  │  w-96 sticky top-0   │  │
│  │   p-6 flex flex-col gap-6│  │  h-screen p-6        │  │
│  │                          │  │  border-l            │  │
│  │  [IntentInput]           │  │                      │  │
│  │  [ChipInput]             │  │  "Preview" label     │  │
│  │  [ChipArea]              │  │  [LivePreview pre]   │  │
│  │  [ClearButton]           │  │  [CopyButton]        │  │
│  └─────────────────────────┘  └──────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

- Controls pane: `flex-1 overflow-y-auto p-6` — scrollable if chips overflow
- Preview pane: `w-96 sticky top-0 h-screen p-6 border-l border-border flex flex-col gap-4` — always visible, never scrolls away (D-09)
- Preview pane width is fixed at `w-96` (384px). Controls pane gets all remaining width via `flex-1`.

### Responsive Collapse (narrow viewports)

Below `md` breakpoint (768px): stack to single column. Preview pane moves below controls pane. `sticky top-0` is removed on mobile (no room to pin). Apply via:

```
<div className="flex flex-col md:flex-row h-auto md:h-screen">
  {/* Controls */}
  <div className="flex-1 p-4 md:p-6 overflow-y-auto">…</div>
  {/* Preview */}
  <div className="w-full md:w-96 border-t md:border-t-0 md:border-l md:sticky md:top-0 md:h-screen p-4 md:p-6">…</div>
</div>
```

## Interaction Contract

### IntentInput

- Element: `<Textarea>` (shadcn/ui)
- Label: "Describe your image" — visible `<label>` element, not placeholder-only
- Placeholder: "e.g. a misty mountain lake at dawn, shot from above"
- Min-height: 96px (approximately 3 lines at body size)
- Max-height: unconstrained (grows with content via resize handle or CSS)
- Behavior: controlled; every keystroke updates Zustand `intent` field → preview re-derives immediately
- Character limit: none enforced in Phase 1 (Zod schema caps at 2000 — silent truncation at that limit only)

### ChipInput

- Element: `<Input>` (shadcn/ui) + `<Button>` (shadcn/ui, `variant="outline"`)
- Label: "Add a style keyword" — visible `<label>`
- Placeholder: "e.g. cinematic, neon, oil painting"
- Submit: Enter key OR clicking Add keyword button
- Dedup: case-sensitive; silently ignore if label already exists in `chips[]`
- Clear on submit: yes — input clears after chip is added
- Disable Add keyword button when input is empty

### ChipArea

- Element: `<Badge variant="secondary">` per chip, with an appended icon-only `<Button size="icon" variant="ghost">`
- Layout: `flex flex-wrap gap-2`
- Chip label: truncated at 40 characters with CSS `truncate` + `max-w-[200px]`; full label visible on hover via `title` attribute
- ✕ button: aria-label `"Remove [chip label]"` — explicit for screen readers
- Touch target: `min-w-[44px] min-h-[44px]` on the ✕ button
- Empty state: no empty state rendering for ChipArea — section is simply absent when `chips.length === 0`

### LivePreview

- Element: `<pre className="flex-1 whitespace-pre-wrap text-sm font-mono break-words">` inside the preview pane
- Content: `serialize(toDraft())` derived on every render — never stored
- Empty state: render the placeholder text "Your prompt will appear here…" in `text-muted-foreground` when preview string is empty
- XSS rule: render as React text child (`{preview}`), never `dangerouslySetInnerHTML` (CLAUDE.md security requirement)

### CopyButton (D-10, locked)

- Element: shadcn/ui `<Button variant="default">` — accent fill, primary CTA
- Placement: bottom of preview pane
- Label — idle: "Copy prompt"
- Label — after success: "Copied!" (flip immediately on successful `clipboard.writeText`)
- Revert delay: 2000ms (2 seconds) then back to "Copy prompt"
- Disabled: when preview string is empty (`!text`)
- Failure (clipboard denied): silent — catch swallows the error. No visible error state in Phase 1. (Phase 5 desktop can add a fallback select-all if needed.)
- Icon: optionally prepend `Clipboard` icon from lucide-react at 16px — acceptable if it aids scannability

### ClearButton + ClearDialog (D-11, locked)

- Element: shadcn/ui `<Button variant="ghost">` for the trigger
- Label: "Clear all"
- Disabled state: render as `disabled` when builder is already empty (no intent text and no chips) — no dialog shown
- When content present: open `<AlertDialog>` (shadcn/ui)
  - Title: "Start over?"
  - Body: "This will clear your intent and all chips. This action cannot be undone."
  - Cancel button: "Keep editing" (`<AlertDialogCancel>`)
  - Confirm button: "Clear everything" (`<AlertDialogAction>`) — styled with `--destructive` background

## Security Contract (from CLAUDE.md)

These constraints are design-level requirements, not implementation notes:

1. **Never use `dangerouslySetInnerHTML` on prompt content.** The preview pane renders
   the serialized prompt string as a React text child only. This applies to chip labels
   and intent text as well.
2. **All user text passes through `sanitize()` before entering the serializer.** The chip
   label is sanitized when added; the intent field is sanitized inside `serialize()`. The
   DOM receives already-sanitized text — XSS is prevented at the data layer, not by
   relying solely on React's escaping.
3. **Chip ✕ button aria-label is constructed from the already-sanitized chip label.**
   Do not interpolate raw user input into `aria-label` attributes without sanitizing first.
