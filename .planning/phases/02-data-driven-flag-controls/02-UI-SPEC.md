# Phase 2 — UI Design Contract

> Visual and interaction contract for Data-Driven Flag Controls.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Inheritance note:** All base tokens (spacing, typography, color, radius, layout) are
> inherited from the Phase 1 contract (`01-UI-SPEC.md`) and confirmed against the
> installed codebase. This document covers only the new flag controls section and any
> Phase 2 specific additions. Where a value matches Phase 1 exactly, it is noted as
> "inherited."

## Spacing Scale

Inherited from Phase 1. No new tokens introduced in Phase 2.

| Token | Value | Tailwind class | Phase 2 usage |
|-------|-------|----------------|---------------|
| xs | 4px | `gap-1`, `p-1` | Icon-to-label gaps within flag rows; AR preset chip internal padding |
| sm | 8px | `gap-2`, `p-2` | Between label row and control within a flag item; AR preset chip gap |
| md | 16px | `gap-4`, `p-4` | Between flag items in the flat list; standard input padding |
| lg | 24px | `gap-6`, `p-6` | Between the flag section and the sections above it (matching existing section gap) |
| xl | 32px | `gap-8`, `p-8` | Not used in Phase 2 |
| 2xl | 48px | `gap-12` | Not used in Phase 2 |
| 3xl | 64px | `gap-16` | Not used in Phase 2 |

**Exceptions:**
- Slider thumb touch target: minimum 44×44px (WCAG 2.5.5 AA). `@base-ui/react/slider`
  handles this natively via its `Thumb` component; do not shrink with CSS overrides.
- Enable toggle (Switch): minimum 44px height in its interactive area via padding.
  The switch itself can be 20×12px visually.
- Version selector and flag clear (×) buttons: `size="icon"` variant at 32px is acceptable
  for desktop; add `min-h-[44px]` on touch targets where these controls are the only
  interactive element in a row.

Inherited from Phase 1. Phase 2 uses the same three roles; no new roles are introduced.

| Role | Size | Weight | Line Height | Tailwind class | Phase 2 usage |
|------|------|--------|-------------|----------------|---------------|
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-relaxed` | Not the primary role in flag controls |

## Color

Inherited from Phase 1. CSS variable names unchanged. Phase 2 extends the accent
reserved-for list.

| Role | CSS variable | Phase 2 usage |
|------|-------------|---------------|
| Dominant (60%) | `--background` | Main app background — unchanged |
| Secondary (30%) | `--card` / `--muted` | Flag section surface — unchanged |
| Accent (10%) | `--primary` | Copy button (Phase 1); active AR preset chip (selected state); enabled Switch thumb (set state) |
| Destructive | `--destructive` | ClearDialog confirm — unchanged; NOT used for flag clear (×) buttons |
| Foreground | `--foreground` | Flag labels, slider value display when set |
| Muted foreground | `--muted-foreground` | Unset flag value display ("—"), placeholder text in seed/--no inputs, dimmed slider track when unset, version placeholder |
| Border | `--border` | Slider track (unset state), flag section divider, input borders |

**Accent extended to (Phase 2):**
1. Copy button — primary CTA (inherited from Phase 1)
2. Active AR preset chip — the selected aspect ratio preset chip uses `variant="default"`
   (accent fill) to show which value is active
3. Enabled Switch thumb — when a flag's enable toggle is ON, the switch thumb uses the
   accent color to signal "this flag will emit"

**Do NOT apply accent to:**
- Slider tracks or indicators (use `--foreground` for the filled portion)
- Flag row hover states
- Version selector hover states
- The × clear buttons

**Unset/disabled visual treatment:**
When a flag's enable toggle is OFF: the control (slider, etc.) renders at `opacity-50`
with `pointer-events-none`. The label row remains full opacity. This makes the label
always readable while signaling the control is inactive without full visual removal.

## Layout Contract

### Flag Section Placement (D-01, D-02 — locked)

The flag controls are a third always-visible section in the left controls pane, stacked
below ChipArea and ClearButton. No collapsible panel, no tabs.

```
Controls pane (flex-1 overflow-y-auto p-4 md:p-6 flex flex-col gap-6)
├── IntentInput          [Phase 1 — unchanged]
├── ChipInput            [Phase 1 — unchanged]
├── ChipArea             [Phase 1 — unchanged]
├── FlagControls         [Phase 2 — NEW]
│   ├── section heading: "Flags"
│   ├── VersionSelect
│   ├── ARControl
│   ├── SliderFlagControl (Stylize)
│   ├── SliderFlagControl (Chaos)
│   ├── NumberFlagControl (Seed)
│   └── TextFlagControl (--no)
└── ClearButton          [Phase 1 — positioned last]
```

The flag section occupies the same `gap-6` (24px) rhythm as other sections. Within the
section, flag items are separated by `gap-4` (16px) in a `flex flex-col`.

### Responsive Behavior (D-03 — inherited)

Identical to Phase 1. Two-pane collapses to single column at `md` breakpoint (768px).
The flag section stacks with the rest of the left pane. No flag-specific responsive
behavior beyond what the pane already provides.

### Flag Item Layout (per flag row)

Each flag item follows this structure:

```
┌──────────────────────────────────────────┐
│ [Flag Label]              [enable toggle] │  ← label row: flex justify-between items-center
│                                           │
│ [control (slider / select / input)]       │  ← control row: full width
│                                           │
│ [value display or validation hint]        │  ← caption row: optional, 12px
└──────────────────────────────────────────┘
```

- Label row: `flex items-center justify-between gap-2`
- Enable toggle (Switch): right-aligned in label row for Stylize and Chaos only (see
  per-flag patterns below)
- Clear button (×): appears inline on the right for Version, AR, Seed, --no — only when
  a value is set

## Interaction Contract

### Flag Section Container (FlagControls)

- Heading: "Flags" — rendered as `<h3 className="text-sm font-medium">` matching section
  heading pattern from Phase 1 ("Preview" label style)
- Layout: `flex flex-col gap-4`
- Renders only the flags supported by `selectedVersionId` (via `getFlagsForVersion()`).
  When a flag is hidden due to version gating, it disappears from the DOM — no disabled
  row, no "unavailable" message (VAL-01 is out of scope).
- When no version is selected: all 5 flags are shown (full list).

### VersionSelect

- Element: shadcn `<Select>` wrapping `@base-ui/react/select`
- Label: "Version" — visible `<label>`, `text-sm font-medium`
- Placeholder: "MJ default (no flag)" — shown when no version is selected (unset per D-04)
- Options: V8.1 (Latest), V7, V6.1, Niji 7, Niji 6 — in this order
- Clear: × icon button (`size="icon" variant="ghost"`) appears to the right of the label
  when a version is selected; click to unset
- On select: stores `VersionDefinition.id` (e.g. `'v7'`), NOT the label string
- Version switch behavior: hide-but-retain (non-destructive). When switching to a version
  that does not support an already-set flag, the flag control disappears from the UI and
  the flag is suppressed from serialization, but the value is retained in state. Switching
  back restores the control with the retained value.

### ARControl

- Layout: label row + preset chips row + custom input row

**Label row:**
- "Aspect Ratio" label left-aligned
- × icon button right-aligned — appears only when any AR value (preset or custom) is set

**Preset chips:**
- 7 chips in a `flex flex-wrap gap-1.5` row: "1:1", "4:5", "3:2", "2:3", "16:9", "9:16", "21:9"
- Each chip: `<Button size="sm">` with `variant="outline"` (unselected) or `variant="default"`
  (selected / accent fill)
- Click unselected chip: sets AR flag to that value; chip becomes `variant="default"`;
  any previously selected preset or custom value is replaced
- Click already-selected preset chip: does NOT deselect (use × in label row to clear).
  This upholds D-06 — clearing is explicit, not by re-clicking.
- Aria: `aria-pressed="true"` on the active preset chip

**Custom input:**
- `<Input>` component, `type="text"`, placeholder "Custom (e.g. 21:9)"
- Full width below the preset chips row
- Validation: `/^\d+:\d+$/` checked on blur and on submit (Enter key)
- If invalid: show caption-style error "Use W:H format, e.g. 21:9" in `text-destructive`
  at 12px below the input
- If valid: set AR flag to the input value; any previously selected preset becomes unselected
- When a custom value is active: no preset chip shows as selected (they are mutually exclusive)
- Custom input is always visible regardless of set/unset state (not gated by a toggle)
- When AR is unset: custom input is empty and no chip is selected

### SliderFlagControl (Stylize and Chaos)

- **Set/unset mechanism:** Switch toggle in the label row (right side).
  - When OFF (unset): slider renders at `opacity-50 pointer-events-none`; value shows "—"
    in `text-muted-foreground`
  - When toggled ON: slider becomes active at the default activation value; value shows the
    number in `text-foreground`
- Default activation value: 100 for Stylize, 0 for Chaos (MJ defaults — still emitted per D-06
  since the flag is now explicitly set)
- Switch aria-label: "Enable [flag label]" (e.g. "Enable Stylize")
- Toggle back to OFF: flag is unset; value retained in state but not emitted
- **Slider element:** shadcn `<Slider>` wrapping `@base-ui/react/slider`
  - `min`, `max`, `step` read from `FlagDefinition.control` — NOT hardcoded in the component
  - Stylize: min=0, max=1000, step=1
  - Chaos: min=0, max=100, step=1
- **Value display:** Caption-style number below or inline-right of the track (e.g. "250")
  — updates live as thumb drags
- `aria-label` on thumb: "[flag label] value" (e.g. "Stylize value")

### NumberFlagControl (Seed)

- **Element:** `<NumberField>` from `@base-ui/react/number-field`
- Layout: increment/decrement buttons flanking a text input (NumberField.Group pattern)
- Placeholder: "random (MJ default)"
- When empty: flag is unset per D-05 (empty field = no value stored; no × needed)
- When a value is entered: flag is set; × icon button appears to the right of the NumberField
  group to clear the field back to unset
- `min`: 0, `max`: 4294967295 — read from `FlagDefinition.control`
- Do NOT use a slider for seed — 4B range makes slider UX impractical (RESEARCH.md pitfall)
- Increment/decrement step: 1 (keyboard up/down arrows; click +/− buttons)
- Large value entry: user types directly into the input field (primary interaction for seed)
- Aria: `aria-label="Seed value"`; increment button: `aria-label="Increase seed"`;
  decrement button: `aria-label="Decrease seed"`

### TextFlagControl (--no)

- **Element:** `<Input>` (existing shadcn component)
- Label: "Exclude (--no)"
- Placeholder: "trees, text, watermark"
- maxLength: 500 (from `FlagDefinition.control.maxLength`)
- When empty: flag is unset
- When text is present: flag is set; × icon button appears inline-right of the input
  (positioned absolutely or as a button sibling in a relative wrapper)
- Click ×: clears input and unsets flag
- Sanitization: value routed through `sanitize()` before storing — not shown to user;
  no inline warning for `--` injection (CLAUDE.md security requirement; silent prevention)
- No character count display in Phase 2 (500 chars is generous enough; counter deferred)
- XSS: render stored value as controlled input value attribute — no `dangerouslySetInnerHTML`

### ClearButton / ClearDialog (updated)

- ClearButton behavior unchanged from Phase 1
- Disabled state update: empty = no intent AND no chips AND no flags set AND no version selected
- ClearDialog body copy updated: "This will clear your intent, all chips, and all flags.
  This action cannot be undone." (see Copywriting Contract)

## Security Contract

These design-level constraints extend Phase 1 security requirements to Phase 2 inputs:

1. **Custom AR input:** Validate with `/^\d+:\d+$/` at the UI level before storing.
   Also route through `sanitize()` as defense-in-depth. Display validation error on
   failure — do not silently corrupt the ratio format.

2. **--no text input:** Route through `sanitize()` before storing. `sanitize()` converts
   `--` sequences to `-`, preventing flag injection. This is silent — do not show the
   sanitized value to the user; show the original input in the controlled field while
   storing the sanitized value.

3. **All flag values rendered in LivePreview or elsewhere:** Render as React text children
   only. No `dangerouslySetInnerHTML` on any user-supplied or flag-derived content.

4. **Version stored as ID not label:** The Select component's value attribute must be
   `VersionDefinition.id` (e.g. `'v7'`), not the display label. Prevents a label-as-ID
   lookup failure in helpers.
